# Checkpoint Format Guide

This document defines the exact format for the Checkpoint Block generated by the `context-compactor` skill.

---

## Structure

```
--- CHECKPOINT (Level X) ---

### Project State
[What is built and confirmed working. Past tense. No speculation.]

### Open Decisions
[Unresolved choices that still affect what comes next. If none, write "None."]

### Next Steps
1. [Most immediate action]
2. [Second action, if known]
3. [Optional third action]

### Key Files
- `path/to/file` — [one-line description of why it's relevant]
- `path/to/other` — [one-line description]

--- END CHECKPOINT ---
```

---

## Writing Rules

- **Project State**: Facts only. What exists in the codebase, what endpoints work, what was fixed. Not what was discussed or planned.
- **Open Decisions**: Things that were left ambiguous or explicitly deferred. A decision is "open" if the next developer needs to make a choice before writing code.
- **Next Steps**: Maximum 3. If there are more, that's a sign the compaction level should be higher.
- **Key Files**: Only files the AI would need to open immediately to continue working. Not a full file list.
- **No timestamps.** No "we discussed..." framing. Dense and factual.
